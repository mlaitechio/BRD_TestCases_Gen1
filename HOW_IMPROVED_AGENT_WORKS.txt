EXAMPLE: ACCOUNT MAPPING REQUIREMENT
================================================================================

OLD AGENT OUTPUT (Generic):
"FR-04: Account Mapping - System should map account numbers"
Acceptance Criteria:
  - Account mapping works
  - Mapping is consistent
[2-3 vague criteria, no specifics]

IMPROVED AGENT OUTPUT (Specific):
"FR-04: Account Number Mapping - System derives deposit bank account based
on ABHFL bank number using master mapping table"
Acceptance Criteria:
  - Deposit account mapping table loaded with all 10 bank codes
  - Bank number 726 maps to account 201001531726
  - Bank number 719 maps to account 201001531719
  - Bank number 757 maps to account 201001531757
  - Bank number 733 maps to account 201001531733
  - Bank number 740 maps to account 201001531740
  - Bank number 512 maps to account 201000294512
  - Bank number 521 maps to account 201000294521
  - Bank number 126 maps to account 200999820126
  - Bank number 275 maps to account 201002601275
  - Bank number 105 maps to account 201002960105
  - Invalid bank numbers rejected with error message
  - Mapping applied consistently across all records
  - Account mapping can be updated via configuration without code changes

  - Mapping table is master data maintained by Finance team
  - Failed mappings block record processing
Compliance Notes:
  - ABHFL bank master is maintained centrally
  - Changes require Finance team approval


================================================================================
EXAMPLE: RETRY LOGIC REQUIREMENT
================================================================================

OLD AGENT OUTPUT (Generic):
"FR-09: Auto Retry - Failed records should be retried"
Acceptance Criteria:
  - Failed records are retried
  - Retry happens automatically
[vague, no details]

IMPROVED AGENT OUTPUT (Specific):
"FR-09: Automatic Retry Mechanism - Failed API transactions are automatically
retriggered at fixed intervals"
Acceptance Criteria:
  - Failed records identified after first API call attempt
  - Retry triggered automatically every 3 hours
  - Retry continues until success or max attempts reached
  - Each retry recorded with timestamp and result
  - Failed records display retry count (0, 1, 2, etc.)
  - Max retry attempts configurable (default: 5)
  - Retry history maintained for audit trail
  - System continues processing new records while retrying old ones
Business Rules:
  - Only records with API failures eligible for auto-retry
  - Validation failures NOT retried automatically
  - 3-hour interval is fixed (cannot be changed by users)
  - Retries stop after 5 attempts or success
Compliance Notes:
  - Retry logic prevents duplicate submissions to A3S LMS
  - All retries logged for audit compliance
